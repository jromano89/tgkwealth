TGK Wealth Demo

One demo story, three moving parts.

The TGK platform brings together two polished demo portals, one reusable backend, and DocuSign IAM and Maestro services. The result is a guided story that feels like a real product experience without turning the demo itself into a large application.

For an audience, not an engineer

The experience is designed to feel simple on the surface: launch the portal, start a workflow, and show what changes across the advisor and investor views.

Reusable behind the scenes

The same backend handles auth, proxying, persistence, and Maestro support so future demos do not need to rebuild the plumbing each time.

Fast to host and easy to move

The frontend stays static and build-free. The backend stays lightweight. That keeps the demo portable across local environments and simple cloud hosting.

Architecture At A Glance

How the experience is organized

The project is intentionally split into clear layers so each part has one job: present the story, support the story, or power the workflow.

Experience Layer
Advisor and Investor Portals

Two lightweight frontends present the same demo from different points of view.

Launcher Select the story and enter the correct portal.
Advisor View Shows client book, agreements, onboarding, and configuration controls.
Investor View Shows accounts, documents, tasks, and embedded workflow follow-up.
Support Layer
Reusable TGK Backend

A single backend keeps the demo coherent by centralizing the services both portals need.

Auth and Consent Handles the DocuSign consent callback and token retrieval.
API Proxy Keeps DocuSign API traffic moving through one controlled backend path.
Demo Data Store Persists employees, customers, envelopes, and tasks in local SQLite.
Workflow Layer
DocuSign IAM and Maestro

Live DocuSign services deliver the embedded workflow moments that make the demo feel real.

IAM Supports consent, tokens, account context, and downstream API access.
Maestro Launches workflow instances such as account opening and asset transfer.
Manifest Bridge Lets the TGK backend present the demo data in a Maestro-friendly shape.
Audience Journey

What someone sees during the demo

The platform is built to support a clean narrative from setup to outcome, not just a list of disconnected pages.

1

Open the story

The presenter starts from the launcher or directly in a portal. Branding, mode, and workflow configuration shape the experience without requiring a rebuild.

2

Trigger a live workflow

The advisor or investor experience launches a real Maestro flow through the backend, using DocuSign services behind the scenes.

3

Show the outcome

The portals refresh around the workflow result so the audience can see how documents, tasks, and customer state evolve across the full story.

Why It Is Built This Way

The design principles behind the demo

This is not trying to be a production platform. It is trying to be a credible, flexible, easy-to-run demo foundation.

Simple frontends

Static portals keep hosting easy and make the demo fast to iterate on.

Reusable backend

Shared plumbing means new demo variations can reuse the same foundation.

Real workflow moments

Live IAM and Maestro integration keeps the story grounded in actual DocuSign capabilities.

Clear demo narrative

The architecture supports a presentation flow that is easy to explain to non-technical audiences.

In plain terms: TGK is a demo shell around real workflow services. The frontends tell the story, the backend keeps the story coordinated, and DocuSign IAM plus Maestro provide the live moments that matter.
